# LinkedIn Draft 19: Role-Centered UX

I keep coming back to this UI/UX idea:

"the user" is too vague for big systems.

A cashier, an operator, a manager, an auditor, and an admin should not inherit the same screen with a few buttons hidden at the end.

The role has to shape the interface earlier.

A cashier needs the current sale state and fast checkout.

An operator needs blocked work, evidence, and safe review actions.

A manager needs fresh numbers and priority.

An auditor needs the event trail.

That changes the design questions.

What is this role responsible for?

What can this role accidentally damage?

Which action should be obvious?

Which action should disappear?

When should the system warn, block, or escalate?

At that point, the screen is carrying part of the system design.

It tells each role what they can safely do next.
